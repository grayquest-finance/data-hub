## Project

A standalone Go service that acts as a **GraphQL aggregation layer** between consumers and our existing backends. It fans out to multiple upstream REST services and merges results into a single GraphQL response.

## Commands

Use `make` as the primary interface (run `make help` to see all targets):

```bash
# Development
make run            # Run locally (reads .env)
make dev            # Live reload via air (install: go install github.com/air-verse/air@latest)
make generate       # Regenerate gqlgen bindings — MUST run after any schema.graphqls change
make build          # Build binary (outputs ./data-hub)

# Testing & Quality
make test           # Run all tests
make test-verbose   # Run tests with verbose output
make test-single TEST=TestFoo PKG=./tests/   # Run a single test
make vet            # go vet
make lint           # golangci-lint (install: brew install golangci-lint)

# Docker
make docker-build   # Build Docker image
make docker-run     # Run container (uses .env)
make up             # Start with docker-compose (API only)
make up-cache       # Start with docker-compose + Redis
make down           # Stop docker-compose services
make stop           # Stop and remove standalone container
make logs           # Tail container logs
make shell          # Open shell in running container

# Monitoring
make health         # curl /health on localhost:8080
make status         # Show container and pod status
make info           # Show environment info

# Cleanup
make clean          # Remove binary, containers, and images
```

> **Important**: Any change to `graph/schema.graphqls` requires `make generate` (runs `go run github.com/99designs/gqlgen generate`) before the server will compile correctly. Skipping this causes "internal system error" at runtime.

## Architecture

### Request flow
```
Consumer → Gin router → APIKeyAuthAllowIntrospection → GraphQL handler (gqlgen)
  → graph/schema.resolvers.go (resolver methods, business logic lives here)
  → client/ (HTTP calls to upstream via base_client.go)
  → upstream REST APIs
```

### Routes (all exposed by `cmd/main.go`)
| Route | Auth | Purpose |
|---|---|---|
| `GET /graphql` | public | gqlgen playground |
| `POST /graphql` | `X-Api-Key` (introspection bypass) | GraphQL queries |
| `GET /health` | public | liveness probe |
| `GET /schema` | public | JSON tree of the schema, recursively expanded from `Query` |

`APIKeyAuthAllowIntrospection` (in `middleware/auth.go`) parses the query and skips the key check when every selection starts with `__` — so clients can introspect without credentials, but real queries still need `X-Api-Key`.

### Key design rules
- `cmd/main.go` — wiring only (logger, config, cache, clients, routes, graceful shutdown). No business logic.
- `graph/resolver.go` — root `Resolver` struct that holds all injected deps (`AdminClient`, `KycClient`, `RepaymentClient`, `Cache`, `Config`, `Logger`).
- `graph/schema.resolvers.go` — gqlgen-managed file. Each resolver method follows the same 4-step pattern: **cache check → upstream call → map upstream → graph model → cache store**. New resolver methods generated by gqlgen are appended to the bottom of this file.
- `graph/helpers.go` — pure mapping/computation helpers shared across resolvers (e.g. `strPtr`, `mapTranchDetails`, `computeRepaymentAndEmi`).
- `graph/generated/` — gqlgen output. **Never edit by hand.**
- `model/upstream/` — raw JSON structs matching upstream responses exactly. `admin.go` covers admin + KYC responses; `repayment.go` covers the repayment service. Never returned to consumers.
- `model/graph/loan.go` — hand-written GraphQL model structs. All fields are pointers so absent data serialises as `null`.
- `model/graph/models_gen.go` — gqlgen output, controlled by `gqlgen.yml`.
- `client/` — one client per upstream service (`AdminClient`, `KycClient`, `RepaymentClient`). All clients embed `BaseClient` (`base_client.go`) which centralises the `gq-api-key` header, JSON marshalling, and request-ID propagation. URLs and keys come from `config.Config` only.
- `config/config.go` — single source of truth. Base URLs/API keys from env vars; endpoint paths hardcoded inside `Load()`. Required env vars fail fast at startup.

### Adding a new field to an existing type
1. Add the field to `graph/schema.graphqls`
2. Add the JSON field to the relevant upstream struct in `model/upstream/{admin,repayment}.go`
3. Add the field to the graph model in `model/graph/loan.go` (pointer types)
4. Run `make generate` (regenerates `graph/generated/` and `model/graph/models_gen.go`)
5. Map the field in the resolver method in `graph/schema.resolvers.go`

### Adding a new upstream endpoint
1. Add the env vars (`*_BASE_URL`, `*_API_KEY`) and a service struct block in `config/config.go`; hardcode the endpoint path in `Load()`
2. Add raw response structs in a new file under `model/upstream/`
3. Add a new client in `client/` that embeds `BaseClient`
4. Inject the client into `graph.Resolver` (in `graph/resolver.go`) and wire it in `cmd/main.go`
5. Extend `graph/schema.graphqls` and run `make generate`
6. Implement the resolver method in `graph/schema.resolvers.go` following the cache → call → map → store pattern; pick a TTL from `cfg.CacheTTL`

## Environment

Copy `.env.example` to `.env`. Required vars cause `config.Load()` to fail fast at startup.

| Variable | Required | Notes |
|---|---|---|
| `PORT` | no | Defaults to `8080` |
| `X_API_KEY` | yes | Validated against incoming `X-Api-Key` header on `POST /graphql` |
| `ADMIN_BASE_URL` / `ADMIN_API_KEY` | yes | Admin service (gqadmin_backend) |
| `KYC_BASE_URL` / `KYC_API_KEY` | yes | KYC service (`svc-kyc`) |
| `SERVICES_BASE_URL` / `SERVICES_API_KEY` | yes | Repayment service (named `SERVICES_*` for historical reasons; powers `RepaymentClient`) |
| `CACHE_ENABLED` | no | `true` to enable Redis (defaults to no-op cache) |
| `REDIS_ADDR` | no | e.g. `localhost:6379` — only used when `CACHE_ENABLED=true` |
| `CACHE_TTL_*` | no | Per-section TTLs in Go duration format (`2h`, `30m`, ...). Defaults in `config/config.go` (e.g. disbursement 2h, payments 5m, KYC 1h) |

In Kubernetes, all secrets are pulled from AWS Secrets Manager secret `data-hub-secret` via ExternalSecrets — no `.env` file is used in deployed environments.

## Upstream API notes

- All upstream services expect `gq-api-key` header (not `X-API-Key`)
- Admin disbursement endpoint uses `application_id` as query param (not `app_id`)
- Repayment endpoint is a POST with `{"app_id": "..."}` body
- Upstream returns HTML error pages (not JSON) on auth failure — if you see `invalid character '<'` errors, the API key or base URL is wrong

## Caching

`cache.NewCache` (`cache/redis.go`) splits read and write enablement:
- If `REDIS_ADDR` is empty → both reads and writes are no-ops.
- If `REDIS_ADDR` is set → **writes always happen** (keeps the cache warm), but reads only happen when `CACHE_ENABLED=true`. Toggling `CACHE_ENABLED=false` therefore forces every request through to upstream while still populating Redis.

Cache keys follow the pattern `gqdg:<applicationID>:<section>` (see resolvers in `graph/schema.resolvers.go`).

## Jenkins CI

The `Jenkinsfile` defines a pipeline triggered against release branches (`release_uat`, `release_staging`, `release_prod`). Key notes:
- Jenkins agent label: `gq_arm_` (ARM build node)
- ECR repo name in Jenkins uses underscores: `data_hub_${TARGET_ENVIRONMENT}` — differs from the Makefile default which uses hyphens (`data-hub-${ENVIRONMENT}`)
- Image tag format: `${BRANCH_NAME}-${COMMIT_ID}` (e.g. `release_uat-abc1234`)
- `DEPLOYMENT_MODE=rollback` skips build/push stages and runs `kubectl rollout undo`

## Docker & Kubernetes

**Docker build note**: The Dockerfile builds for `GOARCH=arm64` and outputs the binary as `api_main` (not `data-hub`). Change `GOARCH` if deploying to x86 hosts.

**Deployment pipeline** (AWS ECR → EKS, `ap-south-1`, account `579897422692`):
```bash
make deploy-pipeline ENVIRONMENT=uat IMAGE_TAG=<tag>   # build → push → deploy
make eks-wait                                           # wait for rollout
make eks-rollback                                       # undo if needed
```
`ENVIRONMENT` maps to EKS cluster: `uat`/`staging` → `sandbox-tools`, `preprod` → `pre-prod-cluster`, `prod` → `production-cluster`.

**Helm chart** (`helm/`): Manages namespace, deployment, service, ingress (AWS ALB), HPA (2–10 replicas), and ExternalSecrets (pulls from AWS Secrets Manager secret `data-hub-secret` via `ClusterSecretStore`). Key values to override per environment: `image.repository`, `image.tag`, `ingress.hosts`, `ingress.annotations.alb.ingress.kubernetes.io/certificate-arn`.
